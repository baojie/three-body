# 用户对 AI 助手的指导汇总

> 基于 33 个会话文件（911 条用户消息）分析整理，2026-04-28

---

## 一、配置文件与系统安全

| 要求 | 严重性 |
|------|--------|
| 任何配置文件修改，必须先备份，再 append-only 写入，禁止整体覆盖 | 🔴 严重 |
| `settings.json` 不能写死绝对路径，因为 git repo 在别人机器路径不同 | 🟡 重要 |
| `.lock` 文件应加入 `.gitignore` | 🟡 重要 |
| 不要加太多无关类型到 `.gitignore` | 🟢 一般 |

---

## 二、并发与锁机制

| 要求 | 严重性 |
|------|--------|
| 多 butler 并发时需要防止竞争：每轮开始前锁定文件，结束后释放 | 🔴 严重 |
| `round_counter.txt` 需使用原子操作防止并发写入 | 🔴 严重 |
| `queue.md` 需加乐观锁：领取任务时写"进行中"，完成后标 `[x]` | 🟡 重要 |
| `/msg` skill 生成的临时文件命名要带随机 ID，防止多 agent 冲突 | 🟡 重要 |

---

## 三、数据格式：recent / history / revision

| 要求 | 严重性 |
|------|--------|
| `recent` 必须为 append-only JSONL 格式，不用整体覆写 JSON | 🔴 严重 |
| 每个 recent diff 针对文件上前一行版本（非 git diff）| 🟡 重要 |
| 每个页面的 `history` 也要改造成 JSONL | 🟡 重要 |
| 改造后所有与 recent 相关的 skill、文档、脚本都要同步更新 | 🟡 重要 |

---

## 四、Butler 工作流程与行为规范

| 要求 | 严重性 |
|------|--------|
| **每轮必须完成足够 WU，一轮只创建一个页面是严重违规** | 🔴 严重（反复纠正） |
| 质量升级（upgrade）不应只改 quality 标签，必须对页面内容实质提升 | 🔴 严重 |
| Dream Round 完成的队列任务要移动到 `done.md`，不能留在 `queue.md` | 🟡 重要 |
| 队列为空时，butler 要自动去原文发现新页面或随机执行 enrich/housekeeping | 🟡 重要 |
| Butler 每轮要留下反思日志 | 🟡 重要 |
| 在固定轮次（W5）要反思是否需要从 shiji wiki 借鉴新功能 | 🟢 一般 |
| 轮次周期调度应使用质数（如 29、37），不能用 33 这样的非质数 | 🟢 一般 |
| 编辑原则文档应在 skill 知识库中，不在 skill 本身里 | 🟢 一般 |
| Skill 的编号规范要与 shiji butler skill 保持一致 | 🟢 一般 |

---

## 五、页面质量与内容写作原则

| 要求 | 严重性 |
|------|--------|
| **Wiki 内容严格限制在三体小说原文范围内，不引入通用条目或一般知识** | 🔴 严重 |
| **原文引文（PN）格式必须从 corpus_search.py 实际搜索结果复制，绝不允许凭空捏造** | 🔴 严重 |
| **所有页面写入必须通过脚本，禁止直接用 Write/Edit 工具操作 `pages/`** | 🔴 严重 |
| 禁止空泛标题和废话（如"XX 与知识守护者的悲剧形象"）| 🔴 严重 |
| 创建页面时要先检查是否已存在同名或别名页面，避免重复 | 🟡 重要 |
| 别名（aliases）不能使用 2 字或 1 字的通用词，防止误匹配 | 🟡 重要 |
| Wikify 时绝对不修改原文的任何可见文字，只做链接 | 🟡 重要 |
| 每个实体在每章中只链接第一次出现，不重复链接 | 🟡 重要 |
| K 值应存在 pages 元数据里，不应写在页面内容里 | 🟢 一般 |
| 删除 tag 是丢失信息的行为，不允许轻易删除 | 🟡 重要 |

---

## 六、Skill 自改进机制

| 要求 | 严重性 |
|------|--------|
| 每一轮 beaver 执行都应改进自己一点点，记录改进日志（CHANGELOG.md）| 🟡 重要 |
| 沉淀下来的规则要写到 skill 目录下的 `MEMORY.md` | 🟡 重要 |
| Skill 文件只存放当前规则，MEMORY.md 存放积累的经验 | 🟢 一般 |
| A2wiki 生成三种知识类型：narrative / fact / skill | 🟢 一般 |
| A2wiki 处理时可以自动执行，不需要用户逐步确认 | 🟢 一般 |

---

## 七、/msg 和 /commit skill 设计

| 要求 | 严重性 |
|------|--------|
| `/msg` 写完后写入临时文件并给出 `git commit -F <file>` 命令 | 🟡 重要 |
| 临时文件命名必须带随机 ID，防止多 agent 并发冲突 | 🟡 重要 |
| `/commit` skill 应按内容分组，一组一组让用户确认，不要一次提交全部 | 🟡 重要 |

---

## 八、Bug 根因追查原则

| 要求 | 严重性 |
|------|--------|
| **遇到 bug 不修表面，要找到并根除根因** | 🔴 严重 |
| 连续修改大小不变（如 3480→3499→3480）是无效写入，要找根源 | 🟡 重要 |
| 出现"claude"作者记录时要解释来源，排除幽灵提交 | 🟡 重要 |
| 主页统计数字不更新时，要找到根源并根除，不能只修表面 | 🟡 重要 |

## 九、前端与 UI 规范

| 要求 | 严重性 |

| 搜索框应放在 banner 里面 | 🟢 一般 |

## 反复强调的核心原则（最高频）

1. **每轮 WU 必须达标**（被纠正 5+ 次）：一轮只干一件事是严重违规
